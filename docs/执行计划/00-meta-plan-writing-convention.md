# 00：Plan 编写约定（元计划）

| 字段   | 值                                 |
| ---- | --------------------------------- |
| 状态   | 现行约定                              |
| 创建   | 2026-03-18                        |
| 适用范围 | `docs/plans/` 下所有以两位序号开头的执行/设计类计划 |

***

## 1. 本文件的目的

* **00 级**文档不参与具体功能排期，只说明：**以后在** **`docs/plans/`** **里新开 plan 时要写清什么**，让半年后的读者仍能还原**前因后果**与**如何落地**。

* 具体专题计划从 **01** 起编号（示例：`01-prompt-sandbox-llm-eval-harness.md`）。

***

## 2. 文件命名

| 规则      | 说明                                                                                     |
| ------- | -------------------------------------------------------------------------------------- |
| 前缀      | **两位数字** + **短横线** + **英文 kebab-case slug**，如 `01-prompt-sandbox-llm-eval-harness.md`。 |
| `00`    | 保留给**元约定**（本文件）或「总览/路线图式」的唯一入口；避免与业务 plan 抢编号。                                         |
| `01–99` | 按主题递增；若同一主题多份迭代，可用 `02-xxx-v2.md` 或在同一文件内用**修订记录**追加，避免编号爆炸。                           |
| 扩展名     | `.md`，UTF-8。                                                                           |

***

## 3. 文首元信息表（必填）

每个 plan **开头**用简短表格列出至少：

| 字段            | 是否必填 | 说明                                                         |
| ------------- | ---- | ---------------------------------------------------------- |
| **状态**        | 必填   | 如：草案 / 进行中 / 已搁置 / 已完成；便于目录扫一眼。                            |
| **创建**        | 必填   | 日期 `YYYY-MM-DD`；重大修订可在文末「修订记录」补一行。                         |
| **关联文档**      | 强烈建议 | 链到 PRD、架构、`docs/*.md`、相关 ADR（`.loom/decisions/`）等，避免上下文散失。 |
| **负责人 / 协作方** | 可选   | 多人协作时写明，减少「找谁问」成本。                                         |

***

## 4. 正文建议结构（按顺序）

以下为**推荐模板**；可根据 plan 体量删减，但 **§4.1–4.5 建议始终保留**（§4.5 若无开放问题，可写「当前无」并一句话说明依据）。

### 4.1 背景与动机（为何现在写这个 plan）

**必须写清「前因后果」**，建议包含：

* **产品/技术上下文**：读者不熟悉专题时，用几段话对齐现状（可引用关联文档）。

* **已暴露的问题或机会**：用编号列表列出现象、痛点、触发事件（如某次对话、线上反馈、技术债）。

* **本 plan 要解决的一句话**：单句目标，避免和「方案」混写。

* **边界（非目标）**：明确**刻意不做**什么，防止范围蠕变。

### 4.2 目标与非目标

* **目标**：可验收的条目（表格或列表），尽量 **SMART**（可测、有时序更好）。

* **非目标**：首版不做什么、交给谁/哪份文档，写清楚。

### 4.3 方案概要

* 架构图（可选 **Mermaid**）、模块职责、与现有代码/目录的挂载点。

* 关键决策的**取舍**（为何选 A 不选 B），一两句即可，细节可链到 `.loom/decisions/`。

### 4.4 落地执行计划

* **分阶段**（阶段 0/1/2…），每阶段含 **checkbox 任务**（`- [ ]`），便于在 PR 或 issue 里勾进度。

* 每阶段注明**依赖**（须先完成什么）与**粗略工期**（人日级即可，不必精确）。

### 4.5 待与用户澄清的问题与建议（必填模块）

**目的**：把「仍依赖产品/业务/资源方拍板」的点显式列出，避免执行到一半才发现方向错误；同时给**建议方案**，降低用户决策成本。

**写法要求**：

1. 使用下表（或等价结构），**每条问题一行**；若当前没有开放问题，写一行说明 **「当前无待澄清项」**，并可注明「依据某次对齐日期 / 已写入决策文档」。
2. **必须标注优先级**，并说明**若不澄清会影响什么**（范围、工期、验收、技术路线等）。
3. 优先级推荐三档（可合并表述，但语义要清楚）：

| 优先级          | 含义                                                 | 典型后果（若不澄清）               |
| ------------ | -------------------------------------------------- | ------------------------ |
| **P0 / 阻塞**  | 不澄清则**无法开工**、或会**推翻 plan 主方向**（选错平台、选错用户群、关键依赖不存在） | 返工整段架构、阶段 1 即卡死、验收标准无法定义 |
| **P1 / 高影响** | 不澄清仍可开工，但**极可能大幅返工**或**验收扯皮**                      | 范围错位、接口白做、数据契约对不上、排期失效   |
| **P2 / 中低**  | 不澄清可**按默认假设推进**，后续可改                               | 体验/命名/非关键参数调整，影响局部或二期    |

**推荐表格列**：

| ID | 优先级 | 问题 / 待澄清点 | 若不澄清会影响什么 | 建议（可选） |
| -- | --- | --------- | --------- | ------ |
| Q1 | P0  | …         | …         | …      |

**与「风险」的区别**：**风险**偏已知不确定性 + 缓解；**本节约束**偏「仍须人类/用户输入才能定案」的**开放问题**。澄清结论落地后，可移到「修订记录」或 `.loom/decisions/`，并在本表删除或标记已解决。

### 4.6 数据、安全与合规（若适用）

* Secret、PII、日志留存、沙箱数据从哪来——**只要涉及就写一节**，避免事后踩雷。

### 4.7 风险与缓解

* 表格：**风险 | 缓解**，保持简短。

### 4.8 验收标准

* 「怎样算这份 plan 做完」的**最低定义**，便于关闭 issue / 归档。

### 4.9 后续演进（可选）

* 明确标为二期，避免和首版验收混谈。

### 4.10 修订记录（建议）

* 表格：`日期 | 修订内容`，重大变更必须留痕。

***

## 5. 写作风格

* **语言**：默认中文；专有名词、API、路径、环境变量可保留英文。

* **链接**：仓库内路径用反引号，如 `` `docs/PROMPTS.md` ``。

* **读者**：假设读者 **6 个月后首次打开**，不依赖「当时群里说过」的隐含信息。

***

## 6. 与仓库其他文档的关系

| 文档类型   | 典型位置                          | 与 plan 的分工                                        |
| ------ | ----------------------------- | ------------------------------------------------- |
| 长期路线图  | `docs/ROADMAP.md`             | 方向与主题；**plan 是某一主题的落地拆解**。                        |
| 任务勾选清单 | `docs/IMPLEMENTATION_PLAN.md` | 可与 plan 互链；plan 偏重**设计与分阶段**，避免完全重复时可只链过去。        |
| 架构     | `docs/ARCHITECTURE.md`        | plan 内方案概要引用架构，不在 plan 里大段复制架构全文。                 |
| 决策记录   | `.loom/decisions/*.md`        | \*\*「为什么」\*\*若需长期保留，可沉淀为 decision，plan 里只引用 slug。 |
