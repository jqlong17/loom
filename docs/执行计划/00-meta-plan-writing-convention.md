## 1. 本文件的目的

规定 **`docs/执行计划/`** 内各计划的 **写法、粒度、验收与质量门禁**，保证：

* 任务 **可独立验证**、**可回归**、**可排障**；  
* **测试驱动开发（TDD）** 为 **默认**工作方式（与纯文档/纯配置变更的豁免见 §6）；  
* **任务完成 = 相关自动化测试已通过**（`npm test` / CI），而非「代码合入即完成」。

同时保留 Loom 原有要求：半年后可读的 **前因后果**、**待与用户澄清**（§4.5）、**风险与数据合规** 等章节结构。

**00 级**不参与具体功能排期；专题执行计划从 **01** 起编号（示例：`01-prompt-sandbox-llm-eval-harness.md`）。

## 4. 正文建议结构（按顺序）

以下为推荐模板；**§4.1～§4.5 建议始终保留**（§4.5 若无开放问题，写「当前无」并注依据）。

### 4.1 背景与动机

* **产品/技术上下文**、**已暴露的问题或机会**、**本 plan 一句话目标**、**边界（非目标）**。

### 4.2 目标与非目标

* 目标尽量 **SMART**；非目标写清范围。

### 4.3 方案概要

* 可选 Mermaid；**代码挂载点**（目录/文件）与关键取舍；可链 `.loom/decisions/`。

### 4.4 落地执行计划

* **分阶段** + checkbox；每阶段注明 **依赖** 与 **粗略工期**。  
* **粒度要求**：对「关键 Task」建议按 **§5** 拆行描述（或单独 Task 表）；纯文档阶段可仅保留 checkbox 但须在 PR 中说明 **豁免测试**（§6）。

### 4.5 待与用户澄清的问题与建议（必填）

* 使用表格：**ID | 优先级 | 问题 | 若不澄清的影响 | 建议**；优先级 **P0 / P1 / P2** 含义如下。

| 优先级 | 含义 | 典型后果（若不澄清） |
|--------|------|----------------------|
| **P0 / 阻塞** | 不澄清无法开工或推翻主方向 | 架构返工、验收无法定义 |
| **P1 / 高影响** | 可开工但易大幅返工或验收争议 | 契约错位、排期失效 |
| **P2 / 中低** | 可按默认假设推进 | 局部体验或二期再改 |

* **与「风险」区别**：风险 = 已知不确定 + 缓解；本节 = **须人类/用户拍板** 的开放问题。结论落地后写入修订记录或 decisions，并更新表格。

## 5. 每条任务（Task）建议写清的要素

执行计划中 **关键 Task**（涉及行为或契约变更）建议满足下表；**缺项视为计划质量不合格**（纯文案/配置可注明豁免，见 §6）。

| 要素 | 要求 |
|------|------|
| **Task ID** | 稳定编号（如 `T3.1`），便于 PR/commit 引用。 |
| **目标** | 一句话 **可验证** 结果。 |
| **涉及代码路径** | 预期创建或修改的目录/文件（例：`src/index.ts`、`tests/usecases.test.ts`），禁止仅写「改一下 MCP」。 |
| **依赖** | 前置 Task ID；无则写「无」。 |
| **TDD 顺序** | **先 failing test 再实现**，或与实现 **同一 PR** 但 **CI 必须先绿**；禁止把「先实现后补测」当默认。 |
| **单测（Unit）** | **路径**：如 `tests/xxx.test.ts`；**覆盖**：正常路径、边界、错误输入（及与 schema/契约不一致数据，若适用）。 |
| **集成测（可选）** | 跨模块（如 CLI 与 usecase 同路径不同入口）可用现有 `adapter-consistency` 类测试对齐。 |
| **E2E** | 本仓库以 **Vitest + 临时目录** 为主；若 Task 影响 **CLI 契约** 或 **MCP 对外行为**，须指定 **测试文件路径** 与 **fixture**（§7）。 |
| **日志** | 若 Task 增加运行时日志：注明 **前缀/级别/关键字段**（§8）；无则写「不涉及」。 |
| **完成定义（DoD）** | **必须**包含：与本 Task 相关的 **`npm test` 用例全部通过**；若有约定 E2E/集成用例，一并通过。 |

---

## 6. 测试驱动开发（TDD）门禁

| 规则 | 说明 |
|------|------|
| **默认流程** | **红 → 绿 → 重构**；新行为优先 **失败用例**（或契约/snapshot），再最小实现。 |
| **例外** | 纯配置、纯文案、**无行为变更** 的文档：可标注 **「无行为变更，豁免单测」**，PR 中说明。 |
| **任务完成** | 关联测试在 **默认分支可重复执行**；若已接 CI，**CI 通过** 后方可将 Task 标为完成。 |
| **禁止** | 以「手动点过」作为 **唯一** 验收；手动仅作 **补充**。 |

---

## 7. 测试用例持久化与回归

| 规则 | 说明 |
|------|------|
| **Fixture 入仓** | 黄金数据放在 **`tests/`** 内联、`tests/fixtures/` 或仓库约定目录，**随 git 管理**；禁止依赖个人本机绝对路径。 |
| **临时目录** | 集成测试用 **`fs.mkdtemp`** 等创建/销毁，避免污染开发者工作区。 |
| **快照** | 对稳定输出使用 snapshot 时，PR 中 **说明变更原因**。 |
| **回归** | 执行计划中引用的路径与 **实际测试 import 路径** 保持一致；改契约时同步更新 fixture 与 **迁移说明**。 |

---

## 8. 日志规范（涉及运行时日志的 Task）

Loom 以 **MCP stdio** 为主，不宜向 stdout 打印噪声；若增加 **可诊断日志**（文件、debug 通道或结构化事件）：

| 规则 | 说明 |
|------|------|
| **前缀** | 统一形如 `[loom:weaver]`、`[loom:mcp]`，便于 grep。 |
| **级别** | `error` / `warn` / `info` / `debug` 语义与业界一致。 |
| **结构化** | 关键字段：`operation`、路径（可脱敏）、`durationMs`（可选）、`error.code`（若有）。 |
| **禁止** | 在生产路径打印 **整段用户正文或未脱敏 payload**；需要时 **debug + 截断**。 |
| **执行计划写法** | Task 的「日志」一行至少列 **2 条 info 级** 应出现场景（如：写入成功、lint 失败原因）。 |

---

## 9. 任务独立性

| 规则 | 说明 |
|------|------|
| **可并行** | 无依赖的 Task 应可分给不同人；依赖在计划中 **列表或图示** 标明。 |
| **可验证** | 评审人可 **只跑相关测试子集**（如 `npx vitest run tests/usecases.test.ts`）确认。 |
| **小步** | 单 Task  Prefer **1～2 个 PR**；过大则拆分。 |

---

## 10. 写作风格

* **语言**：默认中文；专有名词、路径、环境变量可保留英文。  
* **链接**：仓库内路径用反引号，如 `` `docs/待整理/PROMPTS.md` ``。  
* **读者**：假设 **6 个月后首次打开**，不依赖口头上下文。

## 11. 与仓库其他文档及目录的对应关系

| 文档类型 | 典型位置 | 与执行计划的分工 |
|----------|----------|------------------|
| 长期路线图 | [`待整理/ROADMAP.md`](../待整理/ROADMAP.md) | 方向与支柱；执行计划是 **主题落地拆解**。 |
| 任务总表 | [`待整理/IMPLEMENTATION_PLAN.md`](../待整理/IMPLEMENTATION_PLAN.md) | 可互链；避免重复时 **只链一处**。 |
| 架构 | [`待整理/ARCHITECTURE.md`](../待整理/ARCHITECTURE.md) | 执行计划只引用要点，不整篇复制。 |
| 决策 | `.loom/decisions/*.md` | 「为什么」沉淀为 ADR，计划内只引用 slug。 |
| 计划草稿 | `docs/计划草稿/` | 中间态；**定稿验收以 `docs/执行计划/` 为准**。 |

**代码与测试目录（以仓库根为基准）**：

```
src/                 # 应用与 MCP 入口、用例适配
tests/               # Vitest 单测与集成测
prompts/             # MCP 提示词（若 Task 涉及）
docs/执行计划/       # 本目录
docs/待整理/         # Roadmap、IMPLEMENTATION_PLAN 等
```

目录若调整，须 **README + 本文件 + 相关执行计划** 择一处为真并互相同步。
