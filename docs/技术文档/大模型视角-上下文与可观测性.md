# 大模型视角：每轮「看到了什么」与如何模拟

本文回答两件事：

1. **从「像 API 一样喂给模型的一串消息」角度**，一轮对话里通常包含哪些块（哪些是 Loom 管的，哪些是 **OpenCode 等 MCP 宿主**管的）。下文以 **OpenCode** 为主举例（你们的主要使用产品）；其它宿主同理，细节以各自实现为准。  
2. **怎样尽量复现或记录**，让你能对照「我以为 AI 知道的」和「实际上下文里有没有」。

> **结论先说清**：**Loom 单独无法导出宿主里「每一轮完整 prompt」**。完整上下文由 **OpenCode（或其它客户端）** 组装；Loom 只能保证：**工具定义、资源（如 `loom-instructions`）、以及各 tool 的入参/返回** 这一截是可描述、可部分落盘。要在 **OpenCode 链路**上拿到接近完整的模型输入，见 **§7.5**（开源可改 `LLM.stream`）。

---

## 1. 模型「看到」的不是 Loom 进程里的内存

在 **OpenCode + MCP（含 Loom）** 场景下，大致分工是：

| 块 | 谁组装 | 模型是否通常能看到 |
|----|--------|----------------------|
| 系统提示 / Agent 规则 | OpenCode（各 agent、`opencode.json`、指令文件等） | ✅ |
| 对话历史（用户话、助手话） | OpenCode（可能压缩 / compaction，视配置） | ✅（不一定全量） |
| **全部 MCP 工具的 JSON Schema + description** | OpenCode 从已连接 MCP 拉取后注入 | ✅（连接后长期占上下文） |
| 工作区上下文（打开的文件、工具读到的片段等） | OpenCode + 工具链 | ✅（视当轮任务而定） |
| **某次 tool 调用的 arguments** | 模型刚生成的 + 宿主转发 | ✅（在该轮推理输入里） |
| **该 tool 的返回（如 `loom_trace` 文本）** | Loom MCP 返回 → OpenCode 塞进下一轮 | ✅（作为 tool 结果消息） |
| `resources/read`（如 `loom-instructions`） | 模型或宿主发起读取，Loom 返回正文 | ✅（读过之后进入后续轮次） |

因此：**「整个系统智能运行」在体验上 = OpenCode 拼的大上下文 + 模型推理 + Loom 等工具返回的事实数据**。

## 2. 用「消息列表」模拟一轮（概念模型）

下面用 **伪协议** 表示「第 N 轮模型输入长什么样」（与具体厂商字段名无关，只表达信息形态）。

### 2.1 会话开始（尚未调 Loom）

模型侧已常驻（示意）：

```text
[system]  …… OpenCode agent / 项目 opencode.json / 指令 ……

[system 或 tool_definitions]  …… 所有已连接 MCP 的工具名、描述、参数 schema（体积可能很大）……
```

**注意**：只要 MCP 连着，**工具列表 + 说明**往往会持续占用上下文，这是「MCP 容易显得吵」的原因之一（Loom 侧已通过读路径有界、`prompts` 文案等缓解 **返回体**，但 **schema 注入** 仍主要由 **OpenCode** 决定）。

### 2.2 用户发一句：「先读一下 Loom 索引」

```text
[user] 先读一下 Loom 索引
```

模型可能输出 **tool 调用**（示意）：

```text
[assistant → tool_use]
  name: loom_index
  arguments: {}
```

**此时模型「看到」的**：上文全部 + 它自己刚要发的 tool 调用（由宿主执行）。

### 2.3 工具执行后（Loom 把结果送回宿主）

宿主把结果包装成 **tool 结果消息** 再交给模型：

```text
[tool_result]
  name: loom_index
  content: |
    ## Mandatory Read Set
    ### Core Concepts …
    ### Recent Memory …
    ### Full Index
    …（此处为 Loom 组装的 Markdown，可能含截断提示）…
```

**下一轮**模型再生成时，**这段文本就在上下文里**，会参与后续推理。

### 2.4 再调 `loom_trace` / `loom_read`

同理：每一轮都是 **「历史消息 + 新的 user/assistant + 新的 tool_result」** 叠上去，直到触达窗口上限后被 **OpenCode**（如 compaction 等）**裁剪或摘要**——**OpenCode 开源**，需要时可读源码或在 §7.5 挂载点打日志验证。

## 3. 模板：手工填写「这一轮模型大致看到了什么」

你可以复制下面模板，**每次复盘一页**，不用追求和字节级 prompt 完全一致，只要抓住 **块级内容** 就有价值。

```markdown
## 轮次 ID / 时间：____

### A. 系统与规则（宿主，以 OpenCode 为主）
- [ ] 是否包含 OpenCode 全局配置 / 当前目录 `opencode.json` / agent 指令等：有 / 无 / 不详
- [ ] 摘要（自己用一句话写）：____

### B. 工具面（MCP）
- [ ] 当前会话连了哪些 MCP（名字）：____
- [ ] 本轮是否新加载了资源：例如 `loom-instructions` 是 / 否

### C. 对话与引用
- [ ] 用户原话（要点）：____
- [ ] 是否附带当前文件/选区：是 / 否（文件名：____）

### D. 本轮 tool 调用（Loom 相关）
| 工具名 | 入参摘要 | 返回摘要（长度/是否截断） |
|--------|----------|---------------------------|
|        |          |                           |

### E. 不确定 / 怀疑点
- 例如：是否可能被历史裁剪掉：____
```

## 4. 用 Loom 自己落盘：能记录什么、不能记录什么

### 4.1 `fullConversationLogging`（`.loomrc.json`）

配置项见 `src/config.ts` 中 `fullConversationLogging`。开启后，MCP 侧会通过 `appendRawConversationRecord` 往 **`.loom/<storageDir>/events-YYYY-MM-DD.jsonl`** 追加记录（见 `src/raw-conversation.ts`）。

**能记到的（Loom 视角）**：

- 每个被包装的 MCP **tool 名**、**入参**、**返回**（或错误），并带 **脱敏与 `maxPayloadChars` 截断**。

**记不到的**：

- 用户的自然语言、助手的自然语言回复全文、系统提示、其它 MCP 的工具调用、**OpenCode 侧**的压缩与裁剪细节（除非你在宿主里额外打日志）。

因此它是 **「Loom 工具链路的飞行记录仪」**，不是 **「模型完整上下文 dump」**。

### 4.2 与 `events.jsonl` 的区别

- **`events.jsonl`**：偏 **产品与索引行为**（如 `knowledge.traced`、`index.query.executed`），不是完整 tool 原文。  
- **`raw_conversations/*.jsonl`**（默认目录名）：在开启 `fullConversationLogging` 时，偏 **tool 输入输出快照**（有截断/脱敏）。

### 4.3 建议怎么用

1. 在**测试项目**里开启 `fullConversationLogging`，跑一小段真实对话。  
2. 打开当天 `events-*.jsonl`，对照模板 **§3** 把 A～D 补全。  
3. 你会清晰看到：**模型在 Loom 这一侧「换到了什么数据」**，而不是「整轮 prompt」。

## 5. 更接近「全量」的做法（工程向，可选）

| 思路 | 说明 |
|------|------|
| **OpenCode 侧落盘** | 开源：在 `LLM.stream` → `streamText` 前写请求快照（§7.5）；或写插件接 `experimental.chat.*` 钩子。 |
| **MCP 代理** | 在 MCP 客户端与真实 server 之间插一层，打印 `tools/call` 与响应（需自写小工具）。 |
| **评测 harness** | 用固定 `messages[]` 脚本调模型 API，自行打印每轮 payload（完全可控；与「当前 OpenCode 会话」是否逐字节一致取决于你是否复用同一套拼装逻辑）。 |

## 7. 「真实」全量上下文日志：有没有可能？沙箱能不能做到？

### 7.1 先把「真实」说清楚

若「真实」指的是：**与 OpenCode 当轮送进 AI SDK / Provider 的请求完全一致（含 system、messages、tools 等）**——

- **单靠 Loom**：**做不到**。Loom 只是 MCP Server，收不到宿主拼好的完整 `messages[]`。  
- **靠 OpenCode**：**可以**。你是开源用户，在 **`packages/opencode/src/session/llm.ts`** 里 `streamText` 调用前落盘即可（§7.5）；若还要对齐 Provider 再变形后的 wire 格式，可再往 `wrapLanguageModel` / `transformParams` 追一层。

若「真实」指的是：**在你自写的另一条循环里**（不经过 OpenCode UI），模型 API 收到的 JSON **逐轮落盘**——

- **也可以**：由你的脚本组装 `messages`、调 Chat Completions（或等价 API），**在发请求前写日志**（评测 harness / 执行计划 01）。

### 7.2 沙箱 / Harness 为什么能解决这一类问题

仓库里 [`docs/执行计划/01-prompt-sandbox-llm-eval-harness.md`](../执行计划/01-prompt-sandbox-llm-eval-harness.md) 说的沙箱，核心是：

- **隔离工作目录**（`LOOM_WORK_DIR`）、**真实 stdio MCP**（Loom）、**可选真实 LLM**、**场景文件可版本化**、**跑完归档产物**。

若在 **Harness 自己的循环**里实现「多轮对话 + tool calling」：

1. 每一轮在调用模型前，把 **`messages` 数组（及 temperature、tools schema 等）** 序列化写入 `run-xxx/turn-N-request.json`（或 jsonl 一行一条）。  
2. 收到模型响应后，再写入 `turn-N-response.json`。  
3. 执行工具后，把 **tool 结果** 按协议追加进 `messages`，进入下一轮。

这样得到的日志，就是 **该沙箱环境下模型「实际看到」的上下文**——**真、全、可复跑**。  
若与 **OpenCode 里点的同一句用户话**对比，是否逐字节一致，取决于你是否复用了同一套 agent、同一 `opencode.json`、同一 MCP 集合与 compaction 策略；**Harness 与 OpenCode UI 是两条可并行维护的观测链**。

### 7.3 和 `fullConversationLogging` 的对比

| 能力 | 记录的是否算「全量模型上下文」 |
|------|-------------------------------|
| Loom `fullConversationLogging` | **否**。主要是 **Loom 相关 tool 的 input/output** 片段。 |
| 自研 Harness 在发请求前写盘 | **是（仅针对该 Harness 使用的这条 API 链路）**。 |
| 用 OpenCode 跑 MCP | **默认可改源码落盘**（§7.5）；也可用插件钩子做部分字段记录。 |
| 用其它闭源 MCP 客户端 | **取决于对方是否暴露请求体**；否则需代理或官方能力。 |

### 7.4 小结（直接回答你的两个问题）

1. **有没有办法把「AI 实际看到的内容」存成一种真实上下文日志？**  
   - **有**。主路径：**OpenCode 开源** → 在 **`LLM.stream` / `streamText` 前** 落盘（§7.5）。  
   - 辅助路径：**评测沙箱 / 自写 Agent 循环**（执行计划 01），在 **每次调用模型 API 之前** 落盘请求体。  
   - **Loom** 仍只能可靠记录 **自身 tool 的 input/output**（`fullConversationLogging`）。

2. **早年说的沙箱环境能不能实现？**  
   - **能**，而且这是工程上 **最干净** 的路径：沙箱里 **你拥有 message 组装权**，日志自然「真实」。  
   - 执行计划 **01** 就是把这条线产品化/脚本化；落地时在 Harness 里加 **「每轮请求全量落盘」** 即可作为明确验收项。

### 7.5 OpenCode：主产品下的「真·上下文日志」

**可以，且这是你们的主路径**：OpenCode 自己组装 `system` + `messages` + `tools` 再交给 Vercel AI SDK 的 `streamText`；**源码开放**，可在本机仓库里直接加日志或插件。

**落地执行计划（Loom 仓库内）**：[`执行计划/03-opencode-context-request-logging.md`](../执行计划/03-opencode-context-request-logging.md) — **状态：已完成（本机 OpenCode fork）**，文首表格与 §4 勾选为准；环境变量 **`OPENCODE_CONTEXT_LOG_DIR`** 开启后，在 `streamText` 前落盘 **JSONL**（`requests.jsonl`）。联调与归档：`npm run test:e2e-opencode`（见 `tests/e2e-opencode-sandbox/README.md`）；离线样例：`npm run demo:opencode-context-log`。下列挂载点说明仍适用于 fork 维护与二期扩展。

**推荐挂载点（按「离最终请求近 → 远」排序）**：

| 方式 | 位置（OpenCode 仓库内） | 能记到什么 |
|------|-------------------------|------------|
| **改核心或本地 fork** | `packages/opencode/src/session/llm.ts` 中 `streamText({ ... })` 调用 **之前** | 即将传入的 **`messages`（已含 system 条）**、**`tools`**、**温度等参数**；这是 **OpenCode → AI SDK** 这一层的「真实输入」。若还要对齐经 Provider 变换后的 wire 格式，需再关注同文件里 `wrapLanguageModel` 的 `transformParams`（部分厂商还会再改一版）。 |
| **插件钩子（少改上游）** | `packages/opencode/src/session/prompt.ts`：`Plugin.trigger("experimental.chat.messages.transform", …)` | 转模型消息 **之前** 的会话消息结构（`msgs`），适合做审计或裁剪，与最终 API 字段仍差一步 `MessageV2.toModelMessages`。 |
| **插件钩子** | `packages/opencode/src/session/llm.ts`：`experimental.chat.system.transform`、`chat.params`、`chat.headers` | 系统提示拼接、采样参数、请求头等，可配合上面一起做结构化日志。 |

**结论**：若你的目标是 **「在 OpenCode + Loom MCP 这条链路上落盘每一轮模型输入」**，在 **`LLM.stream` 里、`streamText` 前写 JSONL** 是最贴近「模型实际吃到什么」的做法。

**可复现演练**：本仓库提供 **隔离沙箱脚本 + 文档**，用真实 OpenCode + 真实 Loom MCP 跑至少一轮对话，见 [OpenCode-Loom-MCP-演练沙箱.md](./OpenCode-Loom-MCP-演练沙箱.md)（`npm run sandbox:opencode`）。

## 8. 一句话记住

**模型每一轮「看到」的 = OpenCode（宿主）拼好的消息列表（含历史与工具定义）+ 本轮 tool 结果；Loom 负责其中与 Loom 工具相关的返回与说明，不负责替宿主导出完整 prompt。**

- **日常**：**§3 模板**复盘 + **`fullConversationLogging`（§4）** 看 Loom tool 段。  
- **主产品 OpenCode 上要「真·全量」**：**§7.5** 在 `LLM.stream` 前落盘；**Harness**（§7.2 / 执行计划 01）作并行对照亦可。
