## 1. 我们此刻的共识（简）

- Loom 是 **Markdown + Git** 的系统记忆层，通过 **CLI + MCP** 暴露同一内核。
- **提示词**（`prompts/`）解决「模型愿不愿、会不会用工具」；**工具形状与返回边界**解决「环境是否帮模型还是害模型」——两者都是产品，不是附属文档。
- 宿主侧优先认清现实：**OpenCode CLI** 等已是真用户路径；不必以「复刻 Cursor」为评测目标，而以 **可脚本、可归档的宿主 + MCP** 为准。

---

## 2. 下一阶段五条主线（意向）

### A. 把 Loom 明确放在「Harness / ACI 记忆层」上打磨

**方向**：不抢宿主编排器的活，专精 **外部记忆与规约落盘** 的接口质量。

**意向动作**（择要实施，顺序可调）：

- 审视 **读路径**：`index → trace → read` 的默认返回是否**有界**、是否避免「一把 dump」；文档与工具说明与 [调研文](../调研/上下文工程-核心观点与对Loom的启发.md) 中「上下文淹没」对策对齐。
- 审视 **写路径**：lint / 索引 / 事件是否构成**紧反馈环**；错误信息是否**可行动**（便于模型下一轮自纠）。
- 可选：在 `threads` 或文档中提供 **「会话进度 / 待验证清单」** 的推荐结构，与长程智能体的 progress / checklist 叙事衔接（不强制 JSON，保持 Loom 的 Markdown 哲学）。

**对照 Roadmap**：贴近 **P1 统一内核**、**P3 治理门禁**、**P4 反馈闭环**。

---

### B. 可重复的评测链（真实模型 + 真实 MCP + 沙箱）

**方向**：提示词 A/B 与「是否调用 weave」等行为，不能长期依赖单次 IDE 手感。

**意向动作**：

- 按 [`执行计划/01`](../执行计划/01-prompt-sandbox-llm-eval-harness.md) 推进：**阶段 0–1 无 LLM 冒烟**优先；再接入 **OpenAI 兼容 Kimi** 等多轮场景。
- **统一**：评测进程起 Loom MCP 时用本仓库 **`npm run build` + `node dist/index.js`**（已与 01 中 Q2 拍板一致）。
- 产出物：`run-manifest`、`.loom` 树、事件/raw 可选——支撑后续指标与提示词迭代。

**对照 Roadmap**：**P4 数据反馈闭环**；与 IMPLEMENTATION_PLAN 中「会话级聚合脚本」天然衔接。

---

### C. 从「有日志」到「能回答一个问题」

**方向**：events / metrics / raw_conversations 已存在，下一阶段重在 **少而硬的问题** 能否被回答。

**意向动作**（与 [`待整理/METRICS.md`](../待整理/METRICS.md) 一致处优先）：

- 固定复盘问题示例：*本周有多少次写入？probe 转化率？doctor 告警趋势？*  
- 推进 **raw conversation → 与最终沉淀条目对齐** 的离线脚本（IMPLEMENTATION_PLAN 已列为缺口）。
- 在 PR 或发版节奏中，保留「**指标影响 + 验证**」习惯（已有模板则强化执行）。

**对照 Roadmap**：**P4** 为主。

---

### D. 检索与图谱的「下一代」——分层索引（与现网共存）

**方向**：全量扫 md 在规模大时成为瓶颈；**Tree（目录式）+ Graph（关系）** 分层是 IMPLEMENTATION_PLAN 已写的技术债与机会。

**意向动作**：

- 以 **契约先行的 L0/L1/L2**（catalog / digest / graph）为里程碑；**保留 legacy 路径** 降级。
- 每步配合 **事件与指标**（如 query 路径、L2 读取次数），避免「换了索引却不可观测」。

**对照 Roadmap**：**P2 记忆语义与图谱**、**P3 门禁**。

---

### E. 接入体验与文档结构

**方向**：降低「装得上、配得对、第一次写入成功」的摩擦；文档目录当前有 **待整理** 与 **执行计划** 分野，避免读者迷路。

**意向动作**：

- OpenCode / Cursor：延续 [`待整理/OPENCODE_HOOKS_LOOM.md`](../待整理/OPENCODE_HOOKS_LOOM.md)、[`待整理/CURSOR_HINTS.md`](../待整理/CURSOR_HINTS.md) 的「可复制片段」策略；评估 **开箱模板**（IMPLEMENTATION_PLAN 缺口项）。
- 择机把稳定文档从 `待整理/` 迁回 `docs/` 根或固定子目录，**本文件所在 `计划草稿/`** 可在定型后迁入 `执行计划/` 或删除。

**对照 Roadmap**：**P5 生态与协作**。

## 3. 优先级建议（非排期）

| 顺序 | 主线 | 理由 |
|------|------|------|
| 1 | **B 评测链** | 没有可重复实验，提示词与工具改动难以证明「更好」。 |
| 2 | **C 指标问题闭环** | 已有数据管道，补上「一问一答」脚本与复盘节奏，成本低、叙事强。 |
| 3 | **A Harness 记忆层** | 与差异化直接相关，可与 B 并行（B 的场景即 A 的验收场景）。 |
| 4 | **D 分层索引** | 工程量大，适合在 A/B/C 跑通一条垂直切片后再集中投入。 |
| 5 | **E 文档与接入** | 持续渗透；发版前集中扫一轮即可。 |

## 4. 刻意延后或非目标（本阶段不展开）

- 自建与 OpenAI 同量级的全栈可观测性、浏览器 MCP（属宿主或其它产品层）。
- 替代 Git 或放弃 Markdown 人类可读主轴。
- 为追新模型而频繁改内核——**接口与数据契约**优先于换模型。
