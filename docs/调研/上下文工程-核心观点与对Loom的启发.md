# 《Harness / 上下文工程》长文：核心观点与对 Loom 的启发

> 对应读后整理，原文见同目录 [`上下文工程公众号文章.md`](./上下文工程公众号文章.md)（含 OpenAI Harness、Anthropic 长程智能体、SWE-agent / ACI、Awesome Agent Harness 等综述）。

---

## 一、核心观点（压缩版）

1. **瓶颈常在「环境」而非模型**  
   同一模型换接口/工具，任务表现可差很多；业界实验（如 SWE-agent）说明：**精心设计的智能体—计算机接口（ACI）** 相对粗糙 shell，能带来量级明显的相对提升。

2. **Harness 不是什么**  
   不单是系统提示词、API 薄封装、评测模板或「带记忆的聊天」；而是模型运行的**整套设计环境**：可调用的工具、信息格式、历史压缩与护栏、跨轮次交接脚手架等。

3. **上下文窗口 ≠ 内存条**  
   窗口更像**当前工作意识**：无关信息会与关键信息抢注意力；海量 grep/cat 会**淹没**推理。好的接口要**限流、摘要、强制收窄查询**。

4. **ACI 的关键设计取向（编码场景）**  
   - 搜索：有上限、可提示细化，避免「越搜越宽、上下文变垃圾场」。  
   - 阅读：合适粒度（如按行段）、有状态、行号等降低模型「算术/数数」负担。  
   - 编辑：**即时反馈**（如 Lint），在错误扩散前截断级联故障。  
   - 长会话：**折叠旧观察**，让近期状态可读。

5. **跨窗口 / 长程任务**  
   Anthropic 类做法：初始化会话搭脚手架（如 `init.sh`、**结构化功能列表**、进度日志 + Git）；后续会话**一次一块**、**干净提交与交接**，避免「半拉子工程 + 下一会话从头猜」。

6. **OpenAI Harness 侧强调**  
   - **仓库为权威知识**：渐进披露（短入口如 `AGENTS.md` + `docs/` 深链），避免单文件巨文档腐烂。  
   - **反馈环质量决定上限**：能看日志/指标/浏览器所见，比只改代码更能验真问题。  
   - **架构约束机械化**（Linter/测试），错误信息**可行动**，而非仅靠人肉 Review 扩 scale。

7. **Awesome Agent Harness 分层直觉**  
   从人类监督、规约、平台、任务运行器、编排、框架/运行时到最底层「写代码的智能体」——**越往上越决定差异化**；执行层更易被商品化。

8. **重复出现的模式**  
   **渐进式披露**、**Git 工作树/隔离**、**规约进仓、机器可读**、**机械化不变量**、**收紧反馈环**。

9. **工程问题重定向**  
   失效时少问「怎么写更好的提示词」，多问：**缺什么信息？缺什么反馈？上下文哪里被污染？哪些约束该机械化而非靠模型自觉？**

10. **一句话**  
    **模型是推理引擎；Harness 决定这引擎在真实任务里能稳定走多远。**

## 二、和 Loom 的天然对齐（我们已经在做的）

| 长文里的概念 | Loom 中的对应 |
|-------------|---------------|
| 仓库作为事实来源、渐进披露 | `.loom/` + Git；`loom_index` → `loom_trace` → `loom_read` |
| 避免「一次灌满上下文」 | 必读集 + 摘要；工具侧引导分步读 |
| 编辑/写入护栏 | `loom_weave` 走 lint、索引重建、可选提交与事件 |
| 结构化记忆（非闲聊堆叠） | `concepts` / `decisions` / `threads` + frontmatter |
| 跨会话交接 | Markdown 文件持久化 + Git 历史，而非仅靠对话窗口 |
| Harness 不仅是提示词 | `prompts/` 管「认知入口」，**工具与数据形态**管「环境」 |

结论：**Loom 的定位非常接近「为编码/协作智能体提供外部记忆与规约落盘」这一层 Harness**；长文是在更大图景里解释**为什么这类东西值得认真工程化**。

## 三、对 Loom 的启发（可产品化思考，非本期必做）

以下按**投入感 / 优先级** loosely 排序，便于 roadmap 讨论。

1. **更明确地当「ACI 一环」来自我要求**  
   工具返回是否默认**有界**（条数、长度、截断说明）？是否在文案里强调「先小后大」？可与 SWE-agent 的「50 条上限」哲学对照，避免辅助工具反成噪声源。

2. **强化「跨会话真相」线索**  
   长文里的 **progress 文件 + 可勾选任务列表**；Loom 已有 decisions/threads，可考虑：  
   - 是否在文档中推荐一种轻量「会话进度 / 待验证清单」模板（甚至示例 `threads/` 结构）；  
   - 与 `loom_probe` 流程如何衔接（澄清后再写入）。

3. **反馈环文案**  
   向用户/集成方说明：仅接模型不够，**doctor / events / metrics / raw_conversations** 一类能力是「让智能体看见自己行为的后果」，与长文「反馈环质量」一致——可在 `docs/ARCHITECTURE.md` 或 README 用一两段挂钩 Harness 叙事。

4. **规约进仓、机器可读**  
   Loom 已用 Markdown + YAML frontmatter；可强调 **链接与标签** 对「智能体导航」的价值（图谱连续），并对照长文「JSON 防随手改」——我们不一定要 JSON 任务单，但要清楚 **Markdown 可被误改** 时的治理策略（deprecate、doctor）。

5. **不盲从的部分**  
   - 不要求 Loom 复制 OpenAI 级可观测性栈或浏览器 MCP；那是宿主/全栈 Harness 的事。  
   - **OpenCode CLI** 等宿主承担编排时，Loom 专注 **记忆层接口质量**（工具契约、提示词版本、沙箱评测见 `docs/plans/01-…`）即可。

---

## 四、给团队的一句落地话

**把 Loom 当成「智能体工作环境」的一块的来做，而不只是「另一个记笔记的 MCP」**：  
工具形状、返回边界、渐进读写的路径、写入时的校验与事件——都是在做长文所说的 **Harness / ACI**，这和单纯迭代提示词不是同一类杠杆。
